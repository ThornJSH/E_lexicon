# LLM API 기반 감성 분석기 전환 개발 계획서

본 계획서는 기존의 로컬 Kiwi.js 형태소 분석기 및 KnuSentiLex 사전을 활용한 단순 단어 매칭 방식을 탈피하고, 문맥 인지 성능이 뛰어난 **Google Gemini LLM API**를 활용한 감성 분석 시스템으로의 전환 및 고도화 계획을 담고 있습니다.

---

## User Review Required

> [!IMPORTANT]
> **API 키 관리 및 보안**
> * 본 앱은 정적 HTML/JS 기반의 클라이언트 사이드 웹 애플리케이션입니다. 
> * 따라서 백엔드 서버 없이 사용자가 자신의 **Gemini API Key**를 웹 화면의 설정 창에서 직접 입력하여 브라우저 로컬 저장소(`localStorage`)에 저장하고 직접 API를 호출하는 방식을 제안합니다.
> * 이 방식은 서버 구축 비용이 들지 않고 개인 키가 노출되지 않아 안전하지만, API 키가 브라우저에 저장되므로 개인 컴퓨터 환경에서 사용하기 적합합니다. 이에 대해 다른 선호하시는 구조(예: 간단한 백엔드 프록시 서버 구축 등)가 있으신지 확인이 필요합니다.

> [!TIP]
> **하이브리드 모드 제공 여부**
> * API 키가 없거나 오프라인 상태일 때는 기존의 **로컬 Kiwi 엔진(사전 매칭 방식)**으로 기본 분석을 수행하고, API 키가 입력되면 **고정밀 LLM 분석 모드**로 자동 업그레이드되도록 구현하면 두 방식의 장점(무료/오프라인 vs 고정밀 문맥 분석)을 모두 취할 수 있습니다. 본 계획서에는 이 하이브리드 지원 방식을 기본 제안으로 포함하였습니다.

## Open Questions

1. **사용할 LLM 모델 설정**:
   * 속도가 빠르고 단어당 분석 단가가 낮아 텍스트 분석에 최적인 **`gemini-2.5-flash`** 모델을 기본값으로 제안합니다. 혹시 더 높은 지능의 `gemini-2.5-flash-lite` 등 다른 모델을 옵션으로 선택할 수 있도록 구현하길 원하시나요?
2. **사전 검색 탭 유지 여부**:
   * LLM API로 전환되면 단어 1:1 매칭 사전은 실질적으로 문장 분석에 쓰이지 않게 됩니다. 다만, 군산대 감성사전의 단어를 찾아보고 점수를 확인하는 **"감성 사전 검색" 탭**은 그대로 로컬 데이터 기반으로 남겨둘지, 아니면 완전히 제거하고 심플한 문장 분석만 남길지 결정이 필요합니다.

## Proposed Changes

### [Web Application Component]

기존의 정적 페이지 구조를 유지하면서 API 설정 및 풍부해진 결과 대시보드를 시각적으로 표현할 수 있도록 화면과 로직을 수정합니다.

#### [MODIFY] [index.html](file:///e:/Lexicon/index.html)
* **설정 패널 추가**: API 키를 입력 및 테스트하고 `localStorage`에 저장할 수 있는 슬라이드 인(Slide-in) 설정 패널 또는 접이식 카드 UI를 추가합니다.
* **결과 카드 레이아웃 확장**: 
  * 기존 단어 점수 외에 LLM이 제공하는 **'종합 감성 요약 설명(Summary)'** 영역을 추가합니다.
  * 감성이 느껴지는 **'핵심 구절 및 문맥적 설명'**을 테이블이나 카드 목록 형태로 렌더링할 수 있는 UI 구조를 설계합니다.

#### [MODIFY] [app.js](file:///e:/Lexicon/app.js)
* **Gemini API 연동 엔진 구현**:
  * 입력된 API 키로 Google Gemini API 엔드포인트(`https://generativelanguage.googleapis.com/v1beta/models/...`)에 직접 요청하는 비동기 함수를 추가합니다.
  * 안정적인 처리를 위해 항상 동일한 JSON 스키마 구조로 결과를 응답받도록 **Structured Outputs (JSON Mode)**를 유도하는 시스템 프롬프트를 설계합니다.
* **프롬프트 엔지니어링**:
  ```markdown
  당신은 한국어 문장을 분석하여 감성을 수치화하고 해석하는 감성 분석 전문가입니다.
  다음 JSON 스키마에 맞추어 문장을 분석해 주세요:
  {
    "totalScore": 0.0, // -2.0(매우 부정)에서 +2.0(매우 긍정) 사이의 감성 종합 점수
    "sentiment": "매우 긍정" | "긍정" | "중립" | "부정" | "매우 부정",
    "summary": "문장의 감성에 대한 1~2줄 요약 설명",
    "keyPhrases": [
      {
        "phrase": "지갑을 잃어버려서",
        "polarity": -1.5,
        "reason": "소유물 상실로 인한 슬픔과 불편함을 나타내는 구절"
      }
    ]
  }
  ```
* **하이브리드 라우팅**:
  * `localStorage`에 API 키가 존재하고 활성화되어 있다면 Gemini API 분석을 수행합니다.
  * 키가 없다면 기존의 로컬 `analyzeSentiment` 함수로 폴백(Fallback) 처리하여 동작합니다.

#### [MODIFY] [style.css](file:///e:/Lexicon/style.css)
* API 설정 입력 필드, API Key 유효성 검사 성공/실패 인디케이터 스타일 정의.
* 문맥 분석 결과 카드 및 핵심 구절(Reason) 툴팁/호버 카드 등 프리미엄 테마에 맞는 CSS 트랜지션 추가.

## Verification Plan

### Manual Verification
1. **API 키 설정 기능 검증**:
   * 설정 창에서 올바르지 않은 API 키 입력 시 에러 메시지가 출력되는지 검증합니다.
   * 올바른 키를 입력하고 저장했을 때 로컬 저장소에 정상 보존되는지 확인합니다.
2. **감성 판별 정확도 검증**:
   * 다음 세 문장을 테스트하여 기존 사전 방식의 오류가 완벽히 해결되는지 확인합니다.
     * `"길을 가다 지갑을 잃어버려서 너무 속상하고 짜증이 난다."` ➡️ `"을 가다"`가 무시되고 `"지갑을 잃어버려서"`, `"속상하고"`, `"짜증이 난다"`만 부정(Negative)으로 올바르게 검출되는지 확인.
     * `"하나도 안 기쁘다"` ➡️ 전체 뉘앙스를 부정으로 인식하여 음수(-) 점수가 나오는지 검증.
     * `"시험을 망쳐서 참 기분이 좋네요."` ➡️ 반어법을 캐치하여 부정으로 분류하는지 검증.
3. **오프라인/무키(No Key) 동작 검증**:
   * API 키를 제거한 상태에서 기존의 Kiwi 기반 분석 기능이 정상 작동하는지 확인합니다.
